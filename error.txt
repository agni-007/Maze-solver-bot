Here is a practical tuning method for your bot.

**Start Values**
Use these as baseline:

```text
Kp = 1.2
Ki = 0.0
Kd = 0.8
Base speed = 120 first, then increase
Search speed = 90-110
Turn threshold = 260 mm
Collision threshold = 140 mm

```

Do not start testing at high speed. First make it stable at `Base speed = 100-120`.

**What Each Value Does**
`Kp`: how strongly bot corrects left/right error.
Higher = sharper correction. Too high = zigzag.

`Ki`: long-term correction.
For your maze, keep this `0.0` usually. It can cause drift buildup and sudden turns.

`Kd`: damping.
Higher = smoother, reduces oscillation. Too high = slow or twitchy response.

`Base speed`: normal running speed.

`Search speed`: speed when side wall is out of range.

`Turn threshold`: front distance where bot prepares to turn.

`Collision threshold`: emergency stop distance.

`Out-of-range limit`: side ToF value above this is treated as `OUT`.

**Scenario 1: Bot Zigzags Left-Right**
Symptoms:
- Left/right PWM keeps jumping.
- Bot snakes through straight track.

Change:
```text
Decrease Kp by 0.2
OR increase Kd by 0.2
```

Example:
```text
Kp 1.2 -> 1.0
Kd 0.8 -> 1.0
```

**Scenario 2: Bot Slowly Drifts Into Wall**
Symptoms:
- Bot does not correct fast enough.
- It leans toward one wall.

Change:
```text
Increase Kp by 0.2
```

Example:
```text
Kp 1.2 -> 1.4
```

If it starts zigzagging after this, reduce speed or increase `Kd`.

**Scenario 3: Bot Reacts Too Late Before Turn**
Symptoms:
- Front wall appears.
- Bot reaches too close before turning/stopping.

Change:
```text
Increase Turn Threshold
Increase Collision Threshold slightly
```

Example:
```text
Turn threshold 260 -> 300
Collision threshold 140 -> 160
```

**Scenario 4: Bot Turns Too Early**
Symptoms:
- It slows/turns before actually reaching bend.
- It becomes hesitant.

Change:
```text
Decrease Turn Threshold
```

Example:
```text
Turn threshold 260 -> 220
```

**Scenario 5: Bot Stops Too Often**
Symptoms:
- Dashboard shows `FRONT STOP` even when there is enough space.

Change:
```text
Decrease Collision Threshold
```

Example:
```text
Collision threshold 140 -> 110
```

But do not go too low. For a fast bot, keep at least `120-150 mm`.

**Scenario 6: Dashboard Shows OUT Too Often**
Symptoms:
- Left or right ToF shows `OUT`.
- Bot enters `SEARCH FWD`.

Change:
```text
Increase Out-of-range limit
```

Example:
```text
Out-of-range limit 450 -> 600
```

For your geometry, centered side readings may be around `130 mm`, but angled sensors can jump high. Try `450`, `600`, `800`.

Only after stable driving:

```text
Increase Base speed by 10 or 15
```

Example:
```text
120 -> 135 -> 150 -> 165 -> 180
```

After every speed increase, retune:
```text
Increase Kd slightly
Increase Turn threshold slightly
```

Fast speed usually needs earlier turning and more damping.

2. Set:

5. If stable, increase speed.
6. Test curved/turn section.
7. Tune turn threshold.
8. Increase speed again.

> doues the battrey dgrading  lead to wifi sidisconnectioj , i mean current getting low , voltage is constenbt since we use a buck converter

Yes. A degraded battery can cause WiFi disconnections even when a multimeter shows constant buck-converter output voltage.

The battery does not normally “send less current.” Instead, its internal resistance increases. When the ESP32 transmits WiFi or the motors accelerate, current demand rises briefly, causing a very short voltage dip. A multimeter may miss these millisecond dips, but the ESP32 can brown out, reset, or lose WiFi.

Check these signs:

- `MazeRobot` network disappears and returns: ESP32 likely reset/browned out.
- Serial monitor prints `Brownout detector was triggered` or startup messages again: confirmed power problem.
- WiFi remains visible but dashboard stops: more likely software/network congestion.
- Problem becomes worse when motors start: electrical noise or supply sag.

Recommended power improvements:

- Use a buck converter rated for at least `2 A`, preferably `3 A`.
- Add `470–1000 µF` electrolytic plus `0.1 µF` ceramic near the ESP32 supply.
- Keep motor power wiring separate from ESP32 wiring as far as possible.
- Connect grounds at one solid common point.
- Add `100 nF` ceramic capacitors across each motor terminal.
- Avoid powering motors through the ESP32 regulator.
- Test with motors disconnected. If WiFi becomes stable, power sag/noise is the cause.

A buck converter maintains voltage only while its input voltage and current capacity remain sufficient. It cannot compensate when an aged battery sags below its operating margin.

If centered readings are consistently different, for example `Left=115`, `Right=145`, PID tuning cannot completely fix it because the software currently assumes equal readings mean centered. First improve sensor angles, or later add a sensor-offset calibration.

**2. Tune Kp**

Use a straight section and start slowly:

```text
Base Speed: 100
Search Speed: 80
Kp: 1.0
Ki: 0
Kd: 0
```

Place the robot about `10–15 mm` away from the center and run it.

- Correction is too weak: increase Kp by `0.5`.
- Robot crosses the center repeatedly: Kp is too high.
- Robot smoothly returns toward center: Kp is close.

Example progression:

```text
1.0 → 1.5 → 2.0 → 2.5
```

Stop increasing when it begins oscillating, then reduce by one step.

**3. Add Kd**

Kd reduces zigzagging but also amplifies noisy ToF readings.

Start with:

```text
Kd: 0.1
```

Then test:

```text
0.1 → 0.2 → 0.3 → 0.4
```

- Still zigzagging: increase Kd.
- Motors suddenly jump when readings change: decrease Kd.
- Robot reacts slowly and feels reluctant to correct: Kd may be too high.

The current `Kd=0.8` is fairly aggressive. Try lower values while learning the response.

**4. Leave Ki At Zero**

Keep `Ki=0` unless the robot has a small, consistent drift after sensor alignment and Kp/Kd tuning.

Then try only:

```text
Ki: 0.01 → 0.02 → 0.03
```

Excessive Ki can cause delayed swinging and wall contact after long straight sections.

**5. Tune Turns Separately**

Start with:

```text
Base Speed: 100
Turn Threshold: 260 mm
Turn Strength: 90
Collision Threshold: 140 mm
```

Approach one corner and watch `Robot Status`.

Expected sequence:

```text
RUN → TURN LEFT/RIGHT → RUN
```

Adjust based on behaviour:

| Behaviour | Adjustment |
|---|---|
| Turn starts too late | Turn Threshold `260 → 280` |
| Turn starts too early | Turn Threshold `260 → 240` |
| Starts turning but goes wide | Turn Strength `90 → 105` |
| Rotates too sharply or overshoots | Turn Strength `90 → 75` |
| Shows `FRONT STOP` at a valid corner | Check side readings; open side is not being identified |
| Wrong turn direction | Check whether left/right sensors or labels are swapped |

PWM mismatch during `TURN LEFT/RIGHT` is intentional. PWM must match only during `SEARCH FWD`.

**6. Increase Speed Last**

After stable operation at `100`:

```text
100 → 130 → 150 → 170 → 180
```

At every speed, complete several corners before increasing again. Higher speed commonly requires:

- Slightly higher turn threshold.
- Slightly higher turn strength.
- Slightly more Kd.
- Lower search speed than base speed.

A reasonable first complete-run target is:

```text
Kp: 1.5–3.0
Ki: 0
Kd: 0.1–0.4
Base Speed: 130
Search Speed: 90
Turn Threshold: 260–300 mm
Turn Strength: 90–110
Collision Threshold: 140 mm
Side Max: 450 mm
```

For the next test, record the centered left/right readings and describe whether the first failure is straight-line zigzagging, late turning, early turning, wrong-direction turning, or incomplete turning. That determines the next single adjustment.

New sequence:

```text
Opening detected
→ confirm for 80 ms
→ APPROACH LEFT/RIGHT with equal PWM
→ CURVE LEFT/RIGHT with gradual PWM difference

```

Defaults:

```text
Straight Before Turn: 350 ms
Turn Ramp Time:       450 ms
Turn Strength:        50 PWM
```

The old `90 PWM` turn value is deliberately ignored. During the curve, the differential is limited so the inner wheel always retains at least 45% forward drive. It can no longer immediately slow almost to zero.

**Calibrating The 8.5 cm Offset**

At `Search Speed = 125`, measure how long the robot takes to travel exactly `8.5 cm` in a straight line.

Set:

```text
Straight Before Turn ≈ measured time - 80 ms
```

Example: if travelling 8.5 cm takes `430 ms`:

```text
Straight Before Turn = 350 ms
```

The extra `80 ms` is already used for opening confirmation.

**Adjustment Scenarios**

- Still hits the inside wall: increase Approach by `50 ms`.
- Reaches too close to the front wall: decrease Approach by `50 ms`.
- Inner wheel remains too slow: reduce Strength `50 → 45 → 40`.
- Turn is too wide: increase Strength by `5`.
- Steering change is still abrupt: increase Ramp `450 → 550 ms`.
- Robot does not rotate enough: decrease Ramp or increase Strength.
- Repeats the same turn: the new `500 ms` cooldown should prevent this.
